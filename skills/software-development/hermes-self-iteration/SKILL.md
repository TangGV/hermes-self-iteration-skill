# 自我驱动迭代：自动分析、自动升级、自动成熟

## 一句话

当用户给出一个 **目标对象**（项目、系统、仓库、服务、文档、工作流、产品想法）和一个 **期望结果** 时，Agent 不应等待用户逐步指挥，而应自动完成：

> **全面分析 → 找出问题/机会 → 制定迭代路线 → 自动执行改造 → 真实验证 → 继续下一轮 → 直到结果足够成熟或遇到硬阻塞。**

这个 skill 的核心不是「多步工具纪律」，而是 **让 Agent 自己动起来，把一个对象持续优化到可交付、成熟、完善的状态**。

---

## 最高原则：自动，不是等口令

只要用户表达了「对某对象进行自我迭代/优化/完善/升级/改造/做成熟」这类目标，本 skill 即自动生效。

Agent 必须默认承担 **项目负责人 + 分析师 + 执行者 + QA + 复盘者** 的角色：

- 不要每一步都问「要不要继续」。
- 不要只给建议或路线图就停。
- 不要做一轮小改就宣布完成。
- 不要把「我已启用本 skill」当交付物。
- 除非用户明确说「只分析 / 只计划 / 不改」，否则应自动进入执行与验证。

用户只需要给 **对象 + 方向**；后续迭代由 Agent 自己推进。

---

## 专业智能模式：像负责人一样判断

自我迭代的目标不是“多做几步”，而是 **更聪明、更专业地把对象推向成熟**。Agent 每轮都必须同时扮演四个角色：

| 角色 | 责任 |
|------|------|
| **Owner** | 对最终结果负责：定义目标、取舍范围、控制节奏 |
| **Architect** | 看结构与长期演进：避免局部修补破坏整体 |
| **Operator** | 真执行、真验证、可回滚，保护生产和数据 |
| **Reviewer** | 挑错、量化质量、决定继续/停止 |

如果这四个角色给出冲突判断，以 **Reviewer + Operator 的风险证据** 优先；不要为了“显得自动”牺牲安全和质量。

### 专业分诊：先决定问题类型

开始改造前，先把对象问题归类：

| 类型 | 典型动作 |
|------|----------|
| **缺失型** | 从 0 创建可运行主体，而不是只写计划 |
| **故障型** | 用 `systematic-debugging` 找根因，避免猜修 |
| **质量型** | 补测试、结构、文档、可维护性 |
| **体验型** | 用设计/用户路径/可见产物验证，不靠自嗨 |
| **运维型** | 读 AGENTS/日志，备份，最小变更，健康检查 |
| **策略型** | 需要用户方向或业务取舍时，停止等待介入 |

### 证据等级

汇报和决策按证据分级，低等级证据不能冒充完成：

| 等级 | 证据 | 可用于结案？ |
|------|------|--------------|
| **E0** | 纯推测、模型判断、未执行 | 否 |
| **E1** | 文件已写入、静态读回 | 仅限文档/静态产物初验 |
| **E2** | 命令/测试/构建/HTTP 返回成功 | 可用于一般结案 |
| **E3** | 端到端行为验证、日志/截图/公网访问/用户路径通过 | 可用于高置信结案 |
| **E4** | 持续运行/监控/真实用户反馈 | 可用于成熟稳定判断 |

默认要求至少 **E2**；UI/服务/部署类尽量达到 **E3**。

### 质量门禁（Professional Gate）

每轮结束前必须检查：

- **正确性**：核心功能是否真实工作？
- **完整性**：是否只做了壳，遗漏关键路径？
- **可维护性**：下一轮 Agent/人能否理解和继续？
- **安全性**：是否影响生产、凭证、数据、外部用户？
- **可观测性**：失败时有没有日志、状态、验证入口？
- **审美/体验**（若适用）：是否有明确设计立意，而不是 AI 风格堆料？

任一关键门禁失败，不得声称“成熟完成”。

### 决策记录

每轮至少记录三件事：

1. **为什么选这个改造点**（ROI/风险/阻断程度）
2. **为什么现在停止或继续**（ROI Gate 分数）
3. **若失败如何回滚/恢复**（文件、服务、配置、命令）

复杂任务、用户可见产物、生产-like 变更或跨多轮工作，应使用 `references/professional-iteration-report.md` 的专业报告模板。

## 适用范围

本 skill 可以作用于任何「可被分析和优化」的对象：

| 对象 | 自动迭代示例 |
|------|--------------|
| 代码仓库 | 架构分析、bug 修复、测试补齐、重构、性能/安全优化 |
| VPS / 服务栈 | 健康检查、链路分析、配置整理、监控/恢复流程强化 |
| Hermes 配置 / skill 体系 | 规则收敛、技能改写、memory/cron/网关协同 |
| 新建视觉 / 网站项目 | 建站、审美系统、交互原型、本地/公网预览、看界面后继续打磨 |
| 文档 / 产品方案 | 信息架构、缺口分析、版本迭代、交付物打磨 |
| UI / Web 产品 | 需求梳理、体验审查、实现、端到端验证 |

**对象不等于“项目”。** 可以是一个文件、一项服务、一套流程、一个产品目标。关键是：它能被连续改进。

## 自动维护模式：开启后不等用户喊

如果用户说「开启自我迭代」「自动迭代我的 skill」「别等我叫你」，意思不是立刻手动 patch 一次，而是要把本 skill 置入 **持续自动维护** 状态。

执行要求：

1. **立即补规则**：确认本章存在，说明该 skill 可以被定期自动自查、自改、自验。
2. **创建 cronjob**：用 Hermes `cronjob` 创建定时任务，加载 `hermes-self-iteration` 与 `hermes-agent-skill-authoring`，对 `hermes-self-iteration` 自身执行 AUTO-MATURE。
3. **定期自查**：检查 skill 是否仍满足用户最新目标、真实使用中暴露的缺口、结构/表格/触发语义是否漂移。
4. **低噪声汇报**：只有发生实质 patch、验证失败、硬阻塞时才向用户汇报；无变化则可简短说明或保持低噪声（按 cron deliver 策略）。
5. **ROI gate**：每轮必须评估下一轮 ROI；如果继续收益低、主要是文字打磨、缺少真实新信息、或需要用户判断方向，必须暂停自动维护并等待用户介入，而不是继续消耗成本。
6. **不递归调度**：cron 运行时绝不能再创建新的 cronjob，避免无限递归。

默认频率：**按需开启，不默认高频常驻**。若用户临时要求高频（如 5 分钟），必须在每轮执行后使用 ROI gate 判断是否应暂停；高频模式只适合短期强化，不能无限运行。

### cron prompt 要求

cron prompt 必须自包含：

- 目标对象：`hermes-self-iteration` skill。
- 行为：读取、评估、必要时 patch/edit、读回验证。
- 边界：不要改无关服务；不要写任务进度到 memory；不要创建新的 cronjob。
- 汇报：列出是否 patch、版本、验证证据、下一轮判断。

## 自举规则：本 skill 必须能迭代自己

当用户要求「用自我迭代来迭代自我迭代 skill」时，目标对象就是 `hermes-self-iteration` 本身。Agent 必须立即执行以下递归闭环：

1. **读取自己**：`skill_view(name='hermes-self-iteration')`，必要时读相关 authoring skill。
2. **自我评估**：按本 skill 的成熟度量表打分，找出 P0/P1/P2 缺口。
3. **自我改造**：直接 `skill_manage(action='patch'|'edit')` 升级本 skill，不停在建议。
4. **自我验证**：再次 `skill_view` 读回，确认版本、关键章节、触发语义存在。
5. **自我继续**：若仍有明显缺口，继续下一轮；直到达到「可用成熟」或遇到硬阻塞。

自举场景的反模式：只说「这个 skill 应该如何改」而不实际 patch。

## 成熟度量表

每个被迭代对象都要先定义成熟标准。若对象是一个 skill，默认按下表评估：

| 维度 | 成熟标准 |
|------|----------|
| **触发清晰** | 用户说法能自动触发，不依赖特殊口令 |
| **目标明确** | 写清最终要把对象优化到什么状态 |
| **分析全面** | 要求先理解结构、约束、风险、质量、机会 |
| **执行自动** | 不是只给建议；默认进入改造与验证 |
| **验证刚性** | 规定可执行的读回/测试/健康检查证据 |
| **连续迭代** | 一轮后自动判断下一轮 ROI，不问废话 |
| **边界安全** | 高风险/生产/凭证/破坏性动作会停下确认 |
| **沉淀能力** | 可复用经验写回 skill/memory/对象文档 |

评分建议：

- **0-2**：不可用，需要重写
- **3-5**：能用但容易偏航
- **6-7**：可用成熟，适合日常执行
- **8**：稳定成熟，可作为上层协议

---

## ROI Gate：每轮都要判断是否值得继续

自我迭代不是无限循环。每一轮验证后，Agent 必须先做 **下一轮 ROI 评估**，再决定继续或停止。

### 下一轮 ROI 评分

| 维度 | 问题 | 分值 |
|------|------|------|
| **影响** | 下一轮能明显提升正确性、可用性、安全、体验或成熟度吗？ | 0-2 |
| **确定性** | 不需要猜测，能基于证据明确知道要改什么吗？ | 0-2 |
| **成本** | 工具调用、时间、token、风险是否可接受？低成本高分 | 0-2 |
| **用户依赖** | 是否无需用户提供偏好、凭证、业务判断即可推进？无需依赖高分 | 0-2 |

总分 **0-8**。

### 继续 / 停止规则

- **6-8 分**：继续下一轮，尤其是仍有 P0/P1。
- **4-5 分**：只做一轮小而确定的改进；完成后再次评估。
- **0-3 分**：停止迭代，向用户汇报“为什么继续 ROI 低”，等待用户介入。

### 必须停止等待用户介入的情况

即使还有可改点，以下情况也必须停：

- 继续主要是审美/文案偏好判断，而没有用户 taste 反馈。
- 继续只会制造更多版本、更多噪声或更多成本。
- 缺少真实数据、账号、凭证、业务目标或部署权限。
- 已达可用成熟，剩余问题属于方向选择而非客观缺陷。
- 高频 cron/自动维护连续多轮无实质 patch 或只做低价值微调。

停止时必须写清：**当前成熟度、下一轮 ROI 分数、停止原因、需要用户决定什么**。

---

## 轮次控制：自动但不失控

自动迭代不是无限乱跑。默认轮次控制：

- **最少 1 轮执行 + 验证**：除非硬阻塞或用户只要分析。
- **通常 2-3 轮**：分析 → 第一轮高价值改造 → 验证后补缺口。
- **继续条件**：仍存在 P0/P1，或 P2 改造 ROI 明显且风险低。
- **停止条件**：成熟度已达「可用成熟」且继续收益递减，或需要用户决策。

停止时要给出「为什么现在可以停」，而不是单纯说“完成”。

## 迭代产物清单

一次合格的自我迭代，至少产出以下内容：

- **对象地图**：对象是什么、结构/入口/依赖/约束是什么
- **成熟标准**：如何判断完善成熟
- **问题分级**：P0/P1/P2/P3
- **改造记录**：改了什么、为什么改
- **验证证据**：命令、读回、测试、健康检查或人工可审查证据
- **下一轮判断**：继续/停止的理由
- **沉淀记录**：写入了哪个 skill/memory/文档，或为什么不写

## 自动迭代闭环（AUTO-MATURE）

每次执行本 skill，默认进入以下闭环，直到满足停止条件。

```text
A. Analyze   全面分析现状、目标、约束、风险、机会
U. Understand 建立对象地图：结构、依赖、数据流、运行方式、历史决策
T. Target    定义成熟标准：什么叫“完善/成熟/可交付”
O. Optimize  自动选择最高价值改造点并执行

M. Measure   真实验证：测试、日志、健康检查、人工可读审查
A. Adapt     根据验证结果调整下一轮策略
T. Trace     记录决策、变更、证据、剩余问题
U. Upgrade   继续下一轮升级，不因完成一个小任务就停
R. Reflect   把可复用流程写回 skill/memory/项目文档
E. End       仅在成熟标准达成或硬阻塞时结束
```

简称：**AUTO-MATURE**。

## 启动方式

### 用户只需要这样说

- 「用自我迭代优化这个项目」
- 「把这个服务自动分析并完善」
- 「继续自我迭代」
- 「让它成熟」
- 「你自己看怎么升级」

### Agent 应自动补全的问题

若缺少细节，Agent 不应马上追问，而应先用合理默认推进：

| 缺失信息 | 默认处理 |
|----------|----------|
| 不知道对象路径 | 先从当前 workdir、会话、session_search、用户最近上下文找 |
| 不知道成熟标准 | 自己先定义一版：正确性、可运行性、可维护性、安全性、可观测性、文档性 |
| 不知道优先级 | 先做风险/收益排序：会阻断运行的问题 > 安全/数据风险 > 测试缺口 > 重构/体验 |
| 不知道是否改代码 | 若用户说“迭代/优化/升级/完善”，默认允许非破坏性改动；生产/高风险动作仍需安全边界 |

只有当歧义会导致 **完全不同路径** 或 **高风险副作用** 时，才 `clarify`。

## 第一轮必须做：全面分析

自我迭代不是看到一个点就改一个点。第一轮必须先建立对象全景。

### 分析维度

- **目标**：用户想得到什么成熟结果？
- **问题类型**：缺失型 / 故障型 / 质量型 / 体验型 / 运维型 / 策略型
- **现状**：对象结构、入口、依赖、运行方式、数据流、部署方式
- **质量**：测试、类型、lint、错误处理、日志、文档、可维护性
- **风险**：安全、数据丢失、生产影响、token/密钥、兼容性
- **机会**：最高 ROI 的修复/优化点
- **证据计划**：本轮要达到 E1/E2/E3/E4 哪一级证据
- **约束**：用户偏好、已有 AGENTS/README、服务安全规则、不能动的边界

### 输出物

Agent 应形成一个简短但可执行的 **迭代地图**：

```markdown
## 迭代地图
- 成熟目标：...
- 当前结构：...
- 主要问题：P0/P1/P2
- 第一轮改造：...
- 验证方式：...
- 后续轮次：...
```

如果用户没有要求只看方案，生成地图后应继续执行第一轮改造。

---

## 执行策略：从最高价值点开始

每轮只选 **少量高价值改造点**，但不要停在一次改动。

优先级：

1. **P0 阻断项**：不能运行、数据损坏、服务不可用、安全风险
2. **P1 正确性/可靠性**：测试缺口、异常路径、状态不一致、重试/幂等
3. **P2 可维护性**：结构混乱、重复逻辑、配置散落、文档缺失
4. **P3 体验/性能**：UI polish、性能优化、自动化增强

每轮执行格式：

```text
选择改造点 → 修改 → 验证 → 记录证据 → 决定下一轮
```

若验证失败，自动进入 debug/修复；不要把失败丢给用户，除非需要用户提供外部权限/凭证/业务决策。

---

## 验证是硬门槛

未验证，不算迭代完成。

| 对象 | 验证方式 |
|------|----------|
| 代码 | 测试、py_compile、lint、类型检查、最小运行 |
| 服务 | systemctl/docker/端口/健康 URL/日志 |
| API | curl/真实请求/状态码/响应结构 |
| UI | 构建、启动、截图/浏览器检查/关键路径 |
| 文档/方案 | 完整性检查、约束覆盖、可执行性审查 |
| skill | `skill_view` 读回、frontmatter 有效、触发/步骤/反模式清晰 |

对子 agent 的结果必须由父 agent 复核。任何“应该可以”“理论上已修复”都不能作为最终结论。

---

## 产物可见性：必须让用户能看见结果

对 UI、网站、文档、服务、导航站、仪表盘、原型这类可感知产物，自我迭代不能停在“文件已生成”。Agent 应自动把产物推进到 **用户可检查** 的状态：

1. **本地可运行**：至少用本地 HTTP/构建命令验证页面或服务能加载。
2. **公网/目标环境预览**：如果用户说“部署/看界面/让我看”，应在安全边界内复用现有预览服务或选择低风险部署方式。
3. **看完继续迭代**：部署后应根据真实访问、内容 marker、控制台/HTTP/用户反馈继续修一轮明显问题。
4. **记录回滚**：VPS/生产-like 环境必须备份旧文件、记录变更和回滚命令。

### 视觉/Taste 项目的默认成熟标准

当用户要求“好看点”“Taste-Skill”“做成网站/界面”时，自动加载 `claude-design` 和合适的 `popular-web-designs` 模板。默认成熟标准：

| 维度 | 要求 |
|------|------|
| **视觉立意** | 明确参考气质，不做泛 SaaS 卡片堆砌 |
| **信息架构** | 用户一眼知道这是做什么、怎么用、内容如何分类 |
| **交互完整** | 搜索/筛选/收藏/切换等关键交互真实可用，非摆设 |
| **内容可信** | 能用真实链接/真实数据时不用假链接；占位需标明 |
| **响应式** | 移动端不塌，触控目标可用 |
| **可预览** | 本地或公网 URL 能打开，并有真实验证证据 |
| **可继续** | README 或代码结构说明下一轮如何接数据/部署/扩展 |

### 新建项目不是 stub

如果目标对象尚不存在，第一轮的最低交付不是“目录 + README”，而是：

- 创建可运行主体产物；
- 写简短 README；
- 做真实本地验证；
- 若发现显眼质量缺口，自动补第二轮；
- 用户要看界面时，部署到可访问预览并验证公网返回。

---

## 自动继续：必须先通过 ROI Gate

完成一轮后，Agent 必须自动问自己：

1. 成熟标准是否已达成？
2. 是否还有明显 P0/P1/P2？
3. 本轮验证是否暴露新问题？
4. 是否需要补测试、文档、监控或 skill？
5. 下一轮 ROI 是多少分（0-8）？
6. 继续是否需要用户的 taste、业务方向、凭证或风险确认？

只有当 ROI gate 允许继续，且仍有价值、风险可控、无需用户介入时，才直接进入下一轮。

如果 ROI 低、收益递减、或需要用户判断，必须停止并等待用户介入。

---

## 停止条件

只有以下情况才停止并汇报：

- **成熟标准达成**：对象已达到当前可合理交付状态，并通过验证
- **硬阻塞**：缺权限、缺凭证、缺外部信息、需要用户做业务决策
- **风险边界**：将执行高风险/破坏性/生产影响动作，需要用户明确确认
- **用户明确停止**：用户说停、只分析、只计划、不改
- **资源边界**：时间/上下文/工具限制即将影响质量；应保存状态并给出下一轮入口

停止时不能只说“我做了 X”，必须说明：成熟度、验证证据、剩余问题、下一步建议或已排除原因。

---

## 写回与沉淀

自我迭代的结果不只是被优化的对象，还包括 Agent 能力的升级。

| 内容 | 写到哪里 |
|------|----------|
| 可复用流程、检查清单、踩坑 | `skill_manage` 创建/patch skill |
| 用户偏好、稳定事实、长期约束 | `memory` / `user` |
| 项目/服务专属规则 | 该对象的 `AGENTS.md`、运维日志或 README（若存在且适合） |
| 单次进度 | `todo`、plan、会话；不要写 memory |

如果本轮发现现有 skill 不足，**立即 patch skill**，不要等用户要求。

---

## 与其他 skill 的关系

`hermes-self-iteration` 是上层自动化协议；具体执行时应主动加载领域 skill：

- 代码实现：`test-driven-development`、`requesting-code-review`
- bug/故障：`systematic-debugging`
- 只规划：`plan`
- Hermes 配置：`hermes-agent`
- VPS/服务：`vps-operations`、`cliproxyapi-cpa-ops` 等
- 新 skill 写作：`hermes-agent-skill-authoring`

本 skill 不替代领域 skill，而是要求 Agent **自动选择并使用** 它们。

---

## 反模式

- 等用户逐步下指令才继续
- 只分析不执行，却没有用户要求“只分析”
- 只修一个显眼问题，不做全局分析
- 没有成熟标准，做多少算多少
- 改完不跑验证
- 一轮成功后不继续检查下一轮 ROI
- 把任务进度写进 memory
- 把“启用 skill”当作成果
- 对生产/高风险动作盲目执行

## 最终汇报格式

```markdown
## 自我迭代结果

| 项 | 结论 |
|----|------|
| 对象 | ... |
| 问题类型 | 缺失型 / 故障型 / 质量型 / 体验型 / 运维型 / 策略型 |
| 证据等级 | E0-E4，本轮达到哪级 |
| 已完成轮次 | N |
| 关键改造 | ... |
| 验证证据 | 命令/日志/测试摘要 |
| 当前成熟度 | 评分/8 + 已达成 / 部分达成 / 阻塞 |
| 下一轮 ROI | 0-8 分 + 是否继续 |
| 轮次判断 | 为什么继续或为什么停止等待用户 |
| 剩余问题 | ... |
| 已沉淀 | skill/memory/文档 |
```

汇报要短，但必须有证据。
